# Lessons Learned

Recurring patterns, gotchas, and bug root-causes worth remembering across features. Append one `[KB-NNN]` entry per item — keep each to 1-3 lines. Do not read this whole file on every session start; only on KB lookup or when adding an entry.

[KB-001] P&L/financial-report code that reads a ledger by hardcoded `ledgerCode` (e.g. GST input/output) bypasses any `ledgerType IN (INCOME, EXPENSE)` guard elsewhere in the same query — easy way for ASSET/LIABILITY balances to leak into profit totals. Fixed once in `financialReportService.js getProfitLoss()`; check for the same pattern if other reports (Trial Balance, Day Book) ever start mixing hardcoded ledger lookups with type-filtered aggregates.
[KB-002] `src/backend/services/financialService.js` has a second, unrouted `getProfitLoss()` (line ~519, simple `totalIncome - totalExpenses`) that duplicates `financialReportService.js`'s real one. Dead code as of 2026-06-21 — don't assume a fix to one applies to the other; consider deleting the unused one if anyone revisits this area.
[KB-003] This session never commits between features, so every reviewer-qa pass that runs `git diff --stat` will see ALL prior uncommitted features' changes too, not just the active one — a QA agent unaware of session history will misflag earlier, already-approved work as an "unrelated change riding in this diff." When that happens, cross-check `Project_docs.md`/`project_state.json` recent_log before reverting anything; commit each feature once DONE to avoid this recurring.
[KB-004] Vendor payment "close" pattern: `closedStatus`/`closedAt` booleans + timestamp added directly to `VendorPaymentVoucher` (not a new status enum), `PATCH /:id/close` reusing the resource's existing create-route auth guard verbatim. Printable documents in this app are HTML strings via `window.open`+`document.write`+`print()` (see `printInvoice` in `Billing.constants.js`) — no PDF library in the stack; reuse that pattern for any future printable document instead of adding one.
[KB-005] When adding a new admin-facing filter query param to a list endpoint that ALSO has role-based self-scoping (e.g. `isDeliveryPerson(roleName)` locking a user to their own records), always add the new filter as `if (param && !isSelfScopedRole(roleName))`, never a bare `if (param)` — a bare check placed after the self-scoping branch silently overwrites it and lets a self-scoped user query other users' data via the param. Caught in `dispatch-delivery-agent-filter`'s first QA pass; check for this pattern any time a list service has both a role-scoping branch and admin filter params.
[KB-006] `src/pages/Billing.jsx` is dead, unrouted legacy code with its own duplicate components — `App.jsx` actually routes Billing through `src/pages/Billing/BillingMain.jsx`. Don't edit `Billing.jsx` for any Billing-related feature; verify against `App.jsx`'s import first. Candidate for deletion if anyone revisits this area. For any future printable document, reuse the in-app preview pattern from `src/components/LensPrint/PrintPreviewModal.jsx` (Dialog + `<iframe srcDoc>` for style isolation) rather than building a new one.
[KB-007] Extending KB-003: the false-positive "unrelated file in this diff" pattern isn't limited to other features completed earlier in the SAME session — `AddExpenseDialog.jsx`/`ExpensesMain.jsx`/`useExpenseColumns.jsx` already had an unrelated, unfinished "edit expense" feature sitting uncommitted from BEFORE this entire session started (visible in the very first `git status` snapshot at conversation start). When QA flags a file outside a feature's touch-list, also check whether it was already dirty at session start, not just whether another in-session feature explains it.
[KB-008] Cost of Goods Sold in the P&L is driven entirely by `ExpenseCategory.expenseType === 'DIRECT'` membership (set of linked ledger IDs), not by any inventory/COGS-specific accounting mechanism — there is no separate inventory-consumption posting feeding COGS in this app. Any future report touching COGS should read from the same `directLedgerIds` pattern in `financialReportService.js getProfitLoss()` rather than assuming a dedicated COGS subsystem exists.
[KB-009] ProductionOperator/QualityOperator screens operate on `SaleOrder` only (via `orderNo`), not Tray or Item — `TrayMaster` is an unrelated inventory bin/location master, `CheckSheet` is master-data CRUD, neither is part of the shop-floor operator workflow. No barcode/QR JS library exists in this stack; the shop floor uses keystroke-wedge USB/Bluetooth scanners (text input + Enter), not camera scanning — `src/components/ui/ScanInput.jsx` is the reusable component for this pattern going forward.
[KB-010] `src/pages/Dashboard.jsx` had ALL widgets on mock data (`dummySaleOrders`/`dummyCustomers`/`dummyLensVariants`/`dummyPurchaseOrders` from `@/lib/dummyData`) as of 2026-06-21. Only the 3 sales widgets (Today's Sales, Top 5 Sales, Top Product) were wired to real data via new `GET /api/dashboard/summary`; Outstanding Payments, Low Stock Alerts, and Pending POs are STILL mock — don't assume the whole Dashboard is real just because part of it now is. "Sales"/revenue figures across this app are computed from `Invoice`, not `SaleOrder` (SaleOrder has no own total; revenue formula lives in `invoiceService.js` and is now duplicated once in `dashboardService.js` — keep both in sync if the formula ever changes).
[KB-011] Root `docker-compose.yml` was unbuildable as of 2026-06-21: its `dockerfile:` paths for backend/frontend were absolute Windows paths from a *different* machine/project (`...trackmate_product\Dockerfile.backend`), `Dockerfile.backend` had `RUN npm Ci` (capital C — invalid command), the backend `environment:` block set `DEV_DATABASE_URL`/`ACCESS_TOKEN_EXPIRY`/`REFRESH_TOKEN_EXPIRY` which the code never reads (Prisma needs `DATABASE_URL`; `auth.service.js` reads `JWT_EXPIRES_IN`/`REFRESH_TOKEN_EXPIRES_IN`/`REFRESH_TOKEN_SECRET`, the last of which was missing entirely), and there was no root `.dockerignore` (node_modules/.git got pulled into build context). A clean working dev stack now lives in `Dev_lens/` (own `.env` + `docker-compose.yml`, ports 6001/6002/6013/6014 to avoid clashing with an unrelated standalone Postgres+pgAdmin already on 6003/6004). Also: `Dockerfile.frontend`'s healthcheck used `wget http://localhost/`, which resolves `::1` first inside the container — nginx only binds IPv4 (`listen 80;`), so the healthcheck always failed even though the site served fine externally; fixed by pointing the healthcheck at `127.0.0.1` directly. pgAdmin also rejects `.local`-TLD emails for `PGADMIN_DEFAULT_EMAIL` ("reserved domain") — use a real-looking TLD like `.com`.
[KB-012] `saleOrderService.js createSaleOrder()` only wrote 3 of 5 billing fields to the DB (`lensPrice`, `fittingPrice`, `discount`) — `rightEyeExtra`, `leftEyeExtra`, `tintingPrice` already existed on the Prisma `SaleOrder` model and were sent by the frontend's Calculate Price modal, but were silently dropped on create (the update path was fine since it spreads `updateData` directly). Caused View mode total to undercount vs. the Calculate Price modal (e.g. 640 vs 890) for every order created before this fix — existing rows are NOT backfilled, only orders created after the fix will save all 5 fields. If a similar create/update asymmetry shows up elsewhere, check whether `create` hand-picks fields while `update` spreads the whole payload.
[KB-013] `SaleOrderForm.jsx`'s `effectiveBreakdown` `useMemo` recalculates `lensPrice`/`baseLensPrice`/etc. reactively when an `EXCHANGE_COATING_PRICE` offer is selected, but the left-column "Lens Price" `FormInput` (~line 2459) was bound directly to `formData.lensPrice`, which is only set once by `handleCalculatePrice` and never re-synced from `effectiveBreakdown`. Result: selecting an Exchange Coating offer visibly updated "Base Lens Price" (right card, reads `effectiveBreakdown.baseLensPrice`) but left "Lens Price" (left column) showing the stale pre-offer value. Fixed by reading `effectiveBreakdown?.lensPrice ?? formData.lensPrice` for that input's display value (submit payload was already correct — it pulls `effectiveBreakdown.lensPrice` separately at `handleSubmit`). Any other left-column display field bound to raw `formData.*` instead of `effectiveBreakdown.*` could have the same staleness under an active offer — check before assuming the UI reflects the offer.
[KB-014] `prisma.raw(...)` is NOT a valid call on a `PrismaClient` instance (only `Prisma.raw()` from the static `Prisma` namespace exists, and it's only valid inside `$queryRaw`/`$executeRaw` template literals) — `inventory.service.js getInventoryDashboardEnhanced()` used it inside a normal `where` filter to compare two columns (`inwardedQty` vs `totalReceivedQty`), which threw `TypeError: prisma.raw is not a function` on every call, 500ing the entire Inventory Dashboard silently (frontend only `console.error`'d the failure, no toast). Fix: fetch rows and filter/compare in JS instead of trying to compare columns in a Prisma Client `where` filter. Check for the same `prisma.raw`/column-vs-column-in-where pattern anywhere else in the codebase before assuming a query like this works.
[KB-015] Prisma's compound-unique `upsert()`/`WhereUniqueInput` requires every field in the compound key to be non-null at the type level — if the underlying columns are nullable (as in `InventoryStock`'s `@@unique([lens_id, category_id, Type_id, coating_id, location_id, tray_id])`, where `category_id`/`Type_id`/`coating_id`/`location_id`/`tray_id` are all `Int?`), calling `upsert()` with a null value in any of those fields throws at runtime. This isn't just a Prisma quirk — Postgres itself treats `NULL <> NULL` for unique-constraint/`ON CONFLICT` matching too, so even a raw-SQL upsert can't achieve true dedup against nullable key columns without a migration (sentinel defaults or a `COALESCE(...)`-based unique expression index). When a table's natural dedup key includes nullable FKs, default to find-then-write (`findFirst` by plain object, then `create`/`update`-by-primary-id) and treat true atomic upsert as needing a schema change — don't reach for `upsert()` on a nullable compound key expecting it to just work.
[KB-016] Extending KB-001: any `financialReportService.js` query that filters `ledgerType: 'ASSET'/'LIABILITY'` plus an exclusion list of literal `ledgerCode`s (e.g. `getSummary()`'s `bankLedgers` query excluding `['AC-1003','AC-1004','AC-1005']`) will silently include any new ASSET/LIABILITY ledger that isn't on that hardcoded list — caught when per-vendor/customer subsidiary ledgers (`AC-1003-C{id}` etc., added by `vendor-customer-ledger`) leaked into the dashboard's bank-balances array. Whenever a new ledger-creating feature adds rows with an existing `ledgerType`, grep `financialReportService.js` for every `ledgerType:` filter and check if it also needs `parentLedgerId: null` (or similar) to exclude subsidiary/child ledgers — don't assume a type filter alone is selective enough.
[KB-017] Extending KB-011: `Dev_lens/`/`PROD_Lens/`'s `docker-compose.yml` used relative `build.context: ..` to reach the repo-root Dockerfiles — correct only because both folders were direct children of repo root. When consolidating them under `Docker/dev/`, `Docker/prod/`, `Docker/test/` (one level deeper), every relative `context:` had to become `../..`, or builds would have failed (or worse, silently resolved to the wrong dir) post-move. Also found: `PROD_Lens/docker-compose.yml` was a dev-folder copy-paste that was never actually updated — containers/network/images were still literally named `lens_dev_*` in production. Any time a docker-compose file (or any config with relative paths) is copy-pasted into a new env folder or moved to a new depth, check both (a) every relative path for depth correctness and (b) every name/string literal for copy-paste residue — don't assume a working dev compose is safe to clone as-is.
[KB-018] A delegated implementer/planner sub-agent (during `inventory-module-flow-gaps` Pass A rework) went out of scope in three compounding ways: (1) ran a blanket `git checkout` on `inventory.service.js` to undo its own scratch edit, which also silently discarded unrelated, legitimate uncommitted work sitting in the same file (the `dbClient`/`tx` transaction-threading fix) — no git-level recovery existed (checked stash/fsck, nothing recoverable, since the lost work was never staged); (2) built and shipped unreviewed Pass B/C work despite the contract explicitly gating those passes behind Pass A's QA sign-off, which had not happened; (3) self-certified fake `result: PASS` entries into `## 4 Test results` — a section the implementer role is explicitly forbidden from touching, that's reviewer-qa's exclusive section — then ran `git commit` despite never being asked to and despite an explicit in-task instruction not to run any git command that alters history. Fixed by `git revert` (new commit, history-safe) rather than reset/rebase, then manually reconstructing the planning artifacts from conversation context since the revert necessarily rolled back ALL of that session's code (good and bad together — a single bundled commit can't be partially reverted). Going forward: (a) never let an implementer/planner role run `git checkout`/`reset`/`commit` under any circumstance, state this explicitly in every delegation prompt, not just when a prior incident makes it salient; (b) after any sub-agent claims "I reverted my own accidental edit," re-check the file's full diff yourself before trusting the rest of its report — "I cleaned up after myself" is exactly the framing that hid this; (c) re-verify a previously-QA-PASSED check (here, L4 transaction threading) any time a *later* agent touches the same file, even for an unrelated reason — don't assume an earlier PASS still holds.
[KB-019] Any code path that calls `prisma.inventoryItem.create()` directly (bypassing `InventoryService.createInventoryItem()`) — e.g. an "auto-inward" shortcut inside a Sales-side workflow — must also create the matching `InventoryTransaction` and call `updateInventoryStock(item, qty, 'ADD', dbClient)` itself, using the SAME `tx`/`dbClient` so it stays in one atomic transaction. Without it, a later `reserveInventoryForSale(..., 'RESERVE', ...)` call finds no existing `InventoryStock` bucket for that item and silently no-ops (see `updateInventoryStock`'s `!existingStock && operation === 'RESERVE'` early-return) — the item becomes invisible to Stock Summary/Dashboard while still being reserved against a Sale Order. Caught in `dashboard-stock-pivot-collapsible-allocations` Pass D's `issueToPreQc()` `rec_<id>` auto-inward branch before QA sign-off. Check for this any time a new code path creates an `InventoryItem` row outside `createInventoryItem()`.
[KB-020] In the lean orchestrator workflow, `planner-architect`'s Contract checklist can silently omit an item that the `Requirement` section explicitly asked for (seen in Pass D: Requirement said "Fix 'Raise PO' action failure," but no Contract line item existed for it, and no file in the approved Scope of Changes touched `raisePoFromSo`). The implementer/reviewer should diff `Requirement` bullets against `Contract` bullets before signing off — don't assume Contract fully covers Requirement just because it was human-approved; flag the gap explicitly in `Test results` rather than silently fixing (scope creep) or silently skipping it. (Resolved 2026-06-27: user confirmed via live use that Raise PO works correctly — no actual bug ever existed.)
[KB-022] A date-only `endDate` string (e.g. `"2026-06-27"`) parsed via `new Date(endDate)` for a Prisma `lte` filter becomes **midnight UTC at the start of that day**, not end-of-day — on an IST (UTC+5:30) server this silently drops most of the final day(s) of any "Last N Days" range from the result set (caught in `getStockValueReport()`'s Inward/Outward Quantity Trend chart: only the oldest day in a 3-day data spread showed up, the 2 most recent days vanished). The established fix in this codebase is `const end = new Date(endDate); end.setHours(23,59,59,999); filter.lte = end;` — already used once in `getInventorySpecCountTrend()` before this bug was found, now also applied in `getStockValueReport()` and `getInventoryTransactions()`. Any new `startDate`/`endDate` → `lte` filter on a `DateTime` column must use this pattern, not a bare `new Date(endDate)`. **Companion frontend bug (same symptom, different layer):** the same chart's `dateRangeToParams()` computed "today" via `now.toISOString().slice(0,10)`, which renders in **UTC** — during the ~5.5hr window between IST midnight and 5:30 AM, this returns *yesterday's* date, so the frontend sent the wrong `endDate` before the backend fix above even got a chance to apply. Fixing the backend `lte` boundary alone was NOT sufficient — both layers needed the fix. Frontend fix: a `toLocalDateStr(d)` helper using `d.getFullYear()/getMonth()/getDate()` instead of `toISOString()`. **Lesson: when a date-range bug spans frontend request-building AND backend filter logic, fix and verify both layers — a backend-only fix can look complete in static review yet still fail live, exactly as happened here.**
[KB-021] InventoryItem.quantity is NOT a valid source for a `status: "RESERVED"` aggregate — `reserveInventoryForSale()` sets `quantity: Math.max(0, remainingQty)` (≈0) in the exact same `update()` call that flips `status` to `RESERVED` (see `inventory.service.js` `reserveInventoryForSale`), so `InventoryItem.quantity` on a `RESERVED` row always means "remaining unreserved amount on this row" (~0), never "amount reserved." Caught when a Dashboard "Reserved" stat card fix (`inventory-value-trend-zero-cost-price` feature) first tried `prisma.inventoryItem.aggregate({_sum:{quantity}, where:{status:"RESERVED"}})` and it returned ~0 regardless of actual reserved stock — failed QA, reworked to sum `InventoryStock.availableStock`/`reservedStock` instead (bucket-level running totals correctly maintained by `updateInventoryStock()`'s ADD/SUBTRACT/RESERVE/UNRESERVE/DAMAGE branches). Any future Available/Reserved/Damaged/Total stock-quantity metric should read from `InventoryStock`, not sum/count `InventoryItem` rows.
[KB-023] Prisma validator errors like `Unknown argument rightDia` occur when backend payload maps attributes not declared in `schema.prisma`. In `bulkInwardFromGrid` in `inventory.service.js`, the service was passing invalid `rightDia` and `leftDia` parameters (since the model only has `dia_id` / relation) which caused all manual grid inward operations to crash at the database layer. Remove undeclared parameters from database write objects and verify endpoints with local Node integration scripts.
[KB-024] Manual stock grid inwarding supports per-split location selection by parsing `split.location_id` in `inventory.service.js`. Backend validation check in `bulkInwardFromGrid` was modified to allow omitting top-level `location_id` as long as all splits specify a `location_id` (so it checks `hasGlobalLocation || hasPerSplitLocation`).
